React试手项目总结

发表于 2018-03-09

这是FCC上的一个练手项目，涉及到React+Redux+React-Redux+异步fetch的问题。这里以项目为轴，将具体的涉及到的问题和概念全部记录。

首先是React。React基本没什么好说的，更多的是类Flux设计模式以及Redux的具体应用。另外就是传递参数的时候，特别是函数类参数的时候 ，要注意很多问题（作用域的问题，函数参数的问题，this添加等）

主要是介绍Redux。

Action

首先，一般的Action就是一个对象字面量。里面包含了一个type(必须）和其他任何你想存放的数据。（但是根据FSA规范，一般只能是4个大类）。而为了方便创造，将一个有参数的函数返回action变成了主要的创建action的方法。这里要注意，因为函数始终是函数，所以有关函数的技巧和方法都能使用，只要最后是返回一个action。其次，可能会有很多样板代码，所以可以写一个创建action的函数来适配一切action creator会方便很多。

而对于action中的异步请求涉及到中间件React thunk。这个中间件允许你dispatch函数而不是一般的对象字面量。所以可以通过在函数中先dispatch一个action，在return 中再返回一个dispatch，达到异步dispatch不同action的目的。

你也可以将一些判断的逻辑放在一个总函数中，再根据情况dispatch对应的action。

Reducer

Reducer是连接Action和Store的中间部分，所以在Reducer中需要有两种接口。

首先Reducer一定是纯函数，也即(prevState,action) => newState的形式。其次一般都会将Reducer拆分，不同的reducer负责不同的state部分。最后用combineReducer合并起来。这里的话要注意，Reducer在此时其实就在定义initialState了（在createStore没有传入初始state参数的时候），key的话就是Reducer的函数名，value就是函数参数state的默认值。

Store

有必要的话要对createStore进行改造，加入中间件，提供初始参数之类的。

Containers

这里需要区别Container和UI组件。Container是带有逻辑的。一般是用connect连接逻辑到组件上，而UI组件只是接受参数的无状态组件。而在Container中要注意生命周期函数，以及具体的需要发送action的函数，将其传给其他组件。

具体的逻辑一般是两个函数：mapStateToProps和mapDispatchToProps.

第一个是将state转化为props传递给container。第二个则是将props与dispatch联系。当state有变化时，UI都会被自动重新渲染。

第一个方法返回的是一个对象，类似props的对象。而第二个返回的则是定义组件如何发送action。它们都可以接受第二个参数，作为ownprops，这样当组件本身参数有变化时，UI组件也会重新渲染。

UI组件

就是比较普通的React组件

Reducer的组织

大多数应用会处理多种数据类型，通常可以分为以下三类：

- 域数据（Domain data）: 应用需要展示、使用或者修改的数据（比如 从服务器检索到的所有 todos

- 应用状态（App state）: 特定于应用某个行为的数据（比如 “Todo #5 是现在选择的状态”，或者 “正在进行一个获取 Todos 的请求”）

- UI 状态（UI state）: 控制 UI 如何展示的数据（比如 “编写 TODO 模型的弹窗现在是展开的”）
